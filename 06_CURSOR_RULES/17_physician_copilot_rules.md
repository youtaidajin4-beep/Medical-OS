# Purpose

新機能の設計・実装・レビュー時に、**パイロット医師（谷口先生）の現場利用**を最優先する判断基準を定義する。

---

# 必須の2観点

新機能を追加・変更するときは、**必ず**次の2点を検証すること。

## 1. 確認時間を減らせるか

- 先生が SOAP・書類を確認・修正する時間を短縮できるか
- 画面遷移・クリック数を増やしていないか
- 診療中の会話を邪魔しないか
- 修正 UI は快適か（編集→コピー→貼り付けが一画面で完結するか）

**減らせない、または増える場合**: 実装を見直すか、ユーザーにスコープ確認する。

## 2. 先生独自の診療ルールを学習・反映できるか

- 紹介状の書き方・判断基準・定型文を設定・保存できるか
- 先生の修正（差分）が履歴として残り、将来の生成品質向上に使えるか
- テンプレートの単語並べではなく、診療文脈を理解した出力になっているか

**学習の土台にならない場合**: 差分保存・ルール注入・プロンプト改善の余地を検討する。

---

# プロダクト判断の優先順位

```
診療終了後の書類一括生成（紹介状・診断書・意見書・処方など）
    ↓
SOAP 下書き（入口であり目的ではない）
    ↓
付加機能（AIエージェント・写真管理・外部連携など）
```

---

# 実装時のチェックリスト

- [ ] この機能は先生の確認時間を減らすか
- [ ] 先生独自ルールを設定または学習に反映できるか
- [ ] 75〜80点の下書きとして出し、残りは先生修正前提か
- [ ] 患者情報の外部送信は最小限か（Constitution 準拠）
- [ ] 医師確認フローを省略していないか
